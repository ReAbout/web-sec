# 业务逻辑漏洞（原理篇）

## 一句话理解
逻辑漏洞不是代码写错了，而是**设计假设错了**：开发者假设用户会按预期顺序、预期次数、预期身份使用系统，攻击者偏不。

## 为什么扫描器扫不到
| 维度 | 代码层漏洞（SQLi/XSS 等） | 逻辑漏洞 |
| --- | --- | --- |
| 特征 | 有明确的语法/payload 特征 | 请求本身完全合法 |
| 检测 | 规则匹配即可 | 需要理解业务语义 |
| 误报 | 相对可控 | 自动化几乎必然误报 |
| 发现方式 | 扫描器 + fuzz | 人工梳理业务流程 |

逻辑漏洞是"人"的主场，这也是为什么 SRC 高分漏洞里逻辑类占比一直很高。

## 根因分类学
### 1. 信任客户端输入
- 金额、数量、角色、状态从请求里直接取值
- 前端 JS 校验被视为"校验"（绕开前端就裸奔）
- 隐藏字段、disabled 控件照样可提交

### 2. 状态机缺陷（步骤跳跃）
- 业务流程是"第 1 步 -> 第 2 步 -> 第 3 步"，但每个接口独立鉴权
- 攻击者直接请求第 3 步，跳过付费/验证环节
- 状态校验应在服务端维护（session 里记"已完成第几步"），而非依赖"用户拿不到后续接口地址"

### 3. 并发假设
- "先查余额再扣款"两步之间存在时间窗
- 单线程思维写出的代码，在并发请求下校验失效
- 防御靠数据库层：事务、行锁、唯一约束，而不是应用层 if-else

### 4. 边界假设
- 金额为正、数量为正、次数有限、时间有序——全是"应该不会有人乱来吧"
- 负数、零、溢出、极大值、类型混淆（字符串 vs 数字）都能打破假设

### 5. 身份与归属假设
- "能访问这个接口的人，就是该操作这个对象的人"
- 这是越权/IDOR 的根因，展开见 [EXP-IDOR](../exp/EXP-IDOR.md)

## 与代码层漏洞的关系
- 逻辑漏洞常与技术漏洞组合放大：越权 + XSS、竞争 + 支付、弱随机 + 密码重置
- 技术漏洞偏"点"，逻辑漏洞偏"线"（流程）——审计时要既能蹲下来看点，也能站起来看线

## 方法论沉淀
测试逻辑漏洞的正确姿势是"角色扮演"：
1. 假设你是一个想占便宜但完全守规矩（不发任何非法报文）的用户
2. 梳理业务流程图，标出每个环节的"信任假设"
3. 逐个假设问：如果我违反它，服务端会拦我吗？

利用手法、工具（Turbo Intruder 等）见 [EXP-Logic](../exp/EXP-Logic.md)。
