# 模板注入 Server Side Template Injection (SSTI)

## 一句话理解
SSTI（Server Side Template Injection）是指用户输入进入服务端模板引擎的渲染上下文，被当成模板语法解释执行，进而读取敏感对象、调用危险方法甚至达到代码执行。

## 与其他注入的区别
- SQL 注入是参数进入 SQL 解释器
- 命令注入是参数进入命令执行环境
- SSTI 是参数进入模板引擎

问题的关键不在“页面上显示了输入”，而在“输入被模板引擎当成表达式或指令执行”。

## 常见场景
- 邮件模板
- 站内消息模板
- CMS 模板自定义
- 后台自定义欢迎词、签名、主题配置
- 报表导出模板
- 错误页、预览页、调试页
- 把用户输入拼接进 `render()`、`render_template_string()`、字符串模板渲染函数

## 成立条件
通常需要满足：
- 应用动态拼接模板字符串
- 用户输入进入模板本身，而不是仅仅作为模板变量
- 模板引擎支持表达式、过滤器、函数、对象访问
- 模板沙箱缺失或可绕过

## 常见危害
- 读取模板上下文中的敏感变量
- 读取配置、环境变量、密钥
- 访问文件系统
- 调用内置对象和危险函数
- 执行系统命令
- 作为进入框架内部对象图的入口，进一步打穿应用

## 快速判断
### 1. 通用探针
很多模板引擎会使用 `{{ }}` 包裹表达式，因此可先用：

```text
{{7*7}}
```

若返回 `49`，说明输入可能被作为模板表达式执行。

### 2. 更多探针思路
- `{{7+7}}`
- `${7*7}`
- `<%= 7*7 %>`
- `#{7*7}`

不同模板引擎的定界符不同，因此应结合回显特征判断。

### 3. 指纹判断
可根据以下特征推测模板引擎：
- 定界符样式
- 报错信息
- 过滤器、内置函数、对象访问语法
- 框架和语言栈

## 典型成因
### 1. 动态拼接模板
高危示例：

```python
template = "Hello " + user_input
return render_template_string(template)
```

### 2. 错把“模板变量”当“模板代码”
安全做法应是把用户输入当变量传入模板，而不是把输入直接拼进模板源码。

### 3. 自定义模板功能
很多业务为了灵活性，允许管理员配置模板片段，这类功能一旦权限边界不清，很容易演变成高危 SSTI。

## 常见模板引擎关注点
### 1. Python
- Jinja2
- Tornado
- Mako

### 2. Java
- Freemarker
- Velocity
- Thymeleaf

### 3. PHP
- Twig
- Smarty

### 4. JavaScript / Node.js
- Handlebars
- EJS
- Pug
- Nunjucks

不同引擎的表达式能力、对象访问方式和沙箱机制差异很大，利用方法不能直接照搬。

## 常见利用路径
### 1. 数学表达式执行
先验证是否存在表达式求值。

### 2. 变量枚举
查看模板上下文里有哪些对象、函数、配置项可用。

### 3. 对象属性遍历
逐步从当前对象进入框架对象图，寻找：
- 配置对象
- 请求对象
- 应用对象
- 类对象
- 内置函数

### 4. 文件读取与命令执行
如果能访问危险对象或函数，可能进一步实现：
- 文件读取
- 环境变量读取
- 命令执行
- 任意代码执行

## 实战排查思路
### 1. 先判断是否是 SSTI，而不是 XSS
如果输入返回的是“计算结果”而不是“原样 HTML 注入”，更偏向 SSTI。

### 2. 先确定语言和框架
语言栈非常重要，因为：
- 同样是 `{{ }}`，引擎可能完全不同
- payload 的对象链和危险点差异很大

### 3. 先求值，再枚举，再利用
建议顺序：
1. 算术测试
2. 错误探针
3. 上下文变量探测
4. 对象访问与链路扩展
5. 危险函数或文件读写

### 4. 关注后台和低频功能
SSTI 很多时候不在主业务入口，而在：
- 后台自定义模板
- 邮件模板预览
- 报表导出
- 主题和页面配置

## 常见误区
### 1. 只过滤花括号
很多模板引擎并不只支持 `{{ }}`，单纯过滤某个定界符没有通用意义。

### 2. 以为 HTML 转义就够了
HTML 转义主要缓解 XSS，对 SSTI 不构成根本防御。

### 3. 依赖黑名单函数过滤
攻击者常常可以通过对象链、过滤器、内置对象绕过表面限制。

## 防御要点
### 1. 不要拼接模板源码
用户输入只能作为模板变量，不应进入模板语法层。

### 2. 使用安全渲染接口
避免使用类似：
- `render_template_string(user_input)`
- `eval` 风格模板渲染
- 动态模板拼接

### 3. 启用沙箱并最小化能力
若业务必须支持自定义模板，应限制：
- 可用函数
- 可访问对象
- 文件系统访问
- 反射和类加载能力

### 4. 做权限隔离
管理员可编辑模板，不代表模板可以拥有系统级执行能力。

### 5. 记录模板变更
对模板内容做审计、留痕和告警，便于发现异常插入。

## 速查清单
- 先找字符串模板渲染、预览、邮件模板、自定义页面配置
- 先用算术表达式判断是否存在模板求值
- 结合语言、框架、报错、定界符识别模板引擎
- 从上下文变量和对象图入手，寻找文件读写和命令执行能力
- 不把 SSTI 与普通回显、XSS、格式化字符串问题混淆
