# OGNL表达式注入

## 一句话理解
OGNL（Object-Graph Navigation Language）注入，是指用户输入进入 OGNL 表达式解析器后，被当成对象图导航和方法调用语句执行，从而访问对象、修改上下文、调用静态方法，最终可能实现远程代码执行。

## 基础理解
OGNL 本质上是一种对象图导航表达式语言，常见能力包括：
- 访问对象属性
- 调用对象方法
- 访问集合元素
- 做条件判断和字符串拼接
- 在特定实现中访问静态方法和类

正因为表达力强，只要把不可信输入直接交给 OGNL 解析，就很容易从“表达式求值”升级成“对象控制”和“代码执行”。

## 常见场景
- Struts2 参数绑定与表达式求值
- XWork / WebWork 相关历史组件
- 旧版 Confluence 相关表达式链路
- 后台自定义规则、模板、搜索过滤器
- 把用户输入拼接进 OGNL 求值接口的自定义业务逻辑

## 常见危害
- 读取应用上下文对象
- 修改框架上下文和访问控制标志
- 调用危险静态方法
- 命令执行
- 作为进一步打穿 Java Web 框架内部对象图的入口

## 常见成因
- 直接把用户输入传给 OGNL 解析器
- 参数绑定链条中存在表达式注入
- 框架默认机制过于强大，开发未意识到输入已进入表达式层
- 业务中自行扩展了表达式求值功能

## 为什么 OGNL 风险高
和普通模板表达式不同，OGNL 的危险点往往在于：
- 表达式上下文对象丰富
- 历史框架中曾暴露 `ValueStack`、`ActionContext` 等高价值对象
- 某些场景可通过修改成员访问控制、排除列表或方法访问限制，突破原有限制

## 高危框架场景
### 1. Struts2
OGNL 与 Struts2 关系最紧密，也是最经典的 OGNL 注入战场。

重点风险点：
- 参数名解析
- 标签属性求值
- 异常消息、重定向、动作映射中的表达式求值
- 历史版本中的安全限制绕过

### 2. Confluence / XWork 生态
部分历史漏洞会通过 WebWork / XWork / OGNL 机制进入表达式求值链，形成远程代码执行。

## Struts2 里的 OGNL 链路
### 1. 为什么 Struts2 容易出事
Struts2 的很多核心机制本身就依赖 OGNL，例如：
- 参数绑定
- 标签属性求值
- `ValueStack` 取值
- 国际化消息与错误处理
- 结果跳转和部分动态配置

也就是说，OGNL 在 Struts2 中不是“边缘功能”，而是框架主干能力之一。  
一旦某个链路把不可信输入送入表达式层，风险就会非常高。

### 2. 常见进入路径
历史上高危点常出现在：
- 请求参数名或参数值进入 OGNL 解析
- 异常消息、错误页、调试页二次求值
- 标签属性、重定向参数、动作映射
- 文件上传异常链、内容类型解析链

### 3. 典型利用思路
很多经典 `Struts2` RCE 并不是“随便一个参数就直接执行命令”，而是：
1. 找到一个会二次求值的入口
2. 进入 OGNL 上下文
3. 修改访问控制或绕过限制
4. 访问危险类和方法
5. 执行命令

## 常见上下文对象
在 OGNL 语境里，真正有价值的不是“表达式语法本身”，而是“表达式能看到什么对象”。

### 1. 高频对象类型
- 请求对象
- 会话对象
- 应用对象
- `ActionContext`
- `ValueStack`
- 运行时类对象
- 类加载器相关对象

### 2. 为什么 `ValueStack` 重要
`ValueStack` 是 Struts2 的核心对象栈，很多参数绑定、标签求值都围绕它展开。  
一旦攻击者能够借表达式访问或操纵它，就可能进一步触达：
- Action 对象
- 上下文 Map
- 请求相关对象
- 框架内部控制字段

### 3. 上下文暴露决定利用深度
有的场景只能访问普通业务对象，风险仍局限在数据读写层。  
有的场景能摸到框架上下文、成员访问控制和运行时类，这时就容易升级为 RCE。

## 常见利用思路
### 1. 先确认是否进入 OGNL 求值
第一步不是直接打命令执行，而是先确认：
- 输入是否被当成表达式解释
- 当前上下文里有哪些对象可访问
- 是否存在方法访问和静态访问限制

### 2. 再看上下文对象
重点关注：
- 请求对象
- 会话对象
- 框架上下文
- 值栈对象
- 类加载器和运行时对象

### 3. 再评估是否能升级为 RCE
如果能访问：
- 静态方法
- `Runtime`
- `ProcessBuilder`
- 反射链

风险会迅速升高。

## 常见排查思路
### 1. 先找 OGNL 解析入口
代码审计重点：
- OGNL 相关依赖
- 表达式求值工具类
- Struts2 / XWork 处理链
- 任何将用户输入解释为“动态表达式”的逻辑

### 2. 再判断用户是否能控制表达式
关键看：
- 用户控制表达式本身
- 还是仅控制普通变量值

只有前者通常才会发展成 OGNL 注入。

### 3. 再看框架版本和限制机制
OGNL 注入高度依赖历史版本行为。  
同样的表达式，在不同 Struts2 版本、不同安全配置下，可用性差异会非常大。

## 常见绕过方向
### 1. 黑名单绕过
历史上很多防护只是限制少数字符、类名或方法名，常被：
- 字符串拼接
- 间接对象访问
- 上下文变量跳转
- 语法变形

绕过。

### 2. 上下文限制绕过
很多经典 OGNL 漏洞利用的关键，不是 payload 本身多复杂，而是先突破：
- 方法访问限制
- 静态方法访问限制
- 黑名单类限制

### 3. 成员访问控制绕过
OGNL 历史利用中，常见关键点包括：
- `memberAccess`
- 静态方法访问开关
- 类排除列表
- 包排除列表

很多经典利用链的本质，就是先把这些限制改松，再去访问危险类。

### 4. 语法与求值路径绕过
常见思路包括：
- 利用不同求值阶段
- 利用错误消息或标签属性的二次解析
- 利用上下文已有对象而不是显式写危险类名
- 利用字符串拼接、属性跳转、间接引用规避规则

### 5. 多层解析链
有些场景不是“一个参数直接进 OGNL”，而是：
- 参数 -> 模板 / 标签 -> OGNL
- 异常处理 -> 消息格式化 -> OGNL
- 重定向 / 动作映射 -> OGNL

这类链条更容易在审计时被忽略。

## 实战审计重点
### 1. 看是否使用 Struts2
如果目标是老 Java Web 系统，先确认是否存在：
- `struts2-core`
- `xwork`
- 相关标签库和拦截器

### 2. 看表达式是否用户可控
例如：
- 参数名
- 重定向参数
- 模板片段
- 错误信息或国际化消息

### 3. 看上下文暴露是否过多
上下文对象越多，可利用面越大。

### 4. 看是否存在二次求值
很多高危漏洞并不发生在“首次解析”，而发生在：
- 错误处理
- 标签渲染
- 国际化消息
- 文件上传异常回显

这里往往比普通参数绑定更危险。

## 历史漏洞脉络
OGNL 注入最值得复盘的并不是某一条 payload，而是 Struts2 安全边界是如何被一步步打穿又一步步收紧的。

### 1. 早期阶段
特点：
- 对 OGNL 能力限制较少
- 更容易通过表达式直接访问危险对象
- 很多漏洞点出现在参数绑定和标签机制

### 2. 中期阶段
特点：
- 框架开始加入黑名单、访问控制和静态方法限制
- 利用开始依赖上下文修改和限制绕过
- 攻击不再只是“直接写 Runtime”

### 3. 后期阶段
特点：
- 漏洞更多出现在特殊功能链路，如文件上传异常处理、消息渲染、命名空间与重定向等边缘路径
- Payload 越来越依赖具体版本、具体上下文和多步绕过

## 高频 Struts2 漏洞脉络
下面这些 `S2` 编号不必逐个硬背 payload，但建议把“入口类型”记住：

### 1. 参数绑定类
例如早期多起 `S2` 问题，本质是请求参数或参数名进入 OGNL 求值链。

### 2. 标签 / 结果渲染类
这类问题往往发生在：
- 标签属性解析
- 重定向参数
- 结果页面取值

### 3. 错误处理 / 异常消息类
这类漏洞的特点是：
- 入口可能很隐蔽
- 触发条件依赖异常路径
- 一旦进入二次求值，危害很高

### 4. 文件上传相关链路
这类问题历史上影响很大，因为：
- 上传功能常见
- 异常消息链复杂
- 容易把攻击面从“业务参数”转移到“框架异常处理”

## 学习和复盘建议
如果要系统掌握 OGNL 注入，建议按这个顺序：
1. 先理解 `ValueStack` 和 `ActionContext`
2. 再理解框架为什么会把参数送进 OGNL
3. 再看访问控制和黑名单是如何被绕过的
4. 最后再按 `S2` 历史漏洞去看不同入口链路

## 防御要点
### 1. 不解析不可信 OGNL 表达式
这是根本原则。

### 2. 升级框架版本
很多经典 OGNL RCE 都与老版本 Struts2 / XWork 的历史问题直接相关。

### 3. 限制表达式能力
若业务必须支持表达式，应限制：
- 方法调用
- 静态方法访问
- 危险类访问
- 上下文对象暴露范围

### 4. 不依赖黑名单
OGNL 绕过空间很大，只做关键字过滤并不可靠。

## 速查清单
- 先确认目标是否使用 Struts2 / XWork / OGNL
- 先判断用户是否能控制“表达式本身”
- 再看上下文对象、方法访问和静态访问限制
- 高优先排查老框架、后台规则功能、模板和重定向链路
- 一旦表达式可控成立，就继续判断能否升级为 RCE
